Zip files extracted on load aren't tracked as known disk paths, which would cause repeated re-detection. Fixing that.


Implemented folder monitoring for new and removed files. Here's what was added:

### Go backend (`backend/files.go`)
- New `ScanFolderChanges(dirPath, knownPaths)` API compares on-disk `.trc3`/`.zip` files against known paths
- Returns added files (with contents) and removed paths

### Monitor logic (`8-1-load-files.ts`)
- Each monitor tick now scans monitored folders (derived from path-based files and tracked known paths)
- **New files** are loaded automatically (including ZIP extraction) and marked with a green highlight
- **Removed files** are highlighted red, then closed after the configured duration
- ZIP deletion also removes extracted entries from the file list
- `monitoredKnownPaths` tracks on-disk paths so ZIP archives aren't re-detected every tick

### UI
- **File list**: green background for added files, red for removed files
- **Header notices** (`3-0-notice-all.tsx`): shows `+N` (green) and/or `-N` (red) during the highlight period
- **Options dialog**: new setting — "Added/removed file highlight duration (ms)" (default: 1000ms, minimum: 100ms)

### How it works
1. Open a folder via the Wails desktop app (path-based files required)
2. Enable monitoring ("Start Monitoring")
3. When files are added to or removed from the folder, the app detects them on the next poll interval
4. Added files appear in the list and are parsed; removed files flash red briefly then disappear

The highlight/notice duration is shared and configurable in **Options → File changes monitor**.

Tracing the monitoring flow to find why folder add/remove detection isn't firing.


Identifying the issue: folder monitoring requires real disk paths from the Go backend, but "Open Folder" and drag-drop use browser file handles without paths, so `collectMonitoredFolders()` stays empty. Routing folder open and Wails drops through Go path-based loading.


The root cause was that folder monitoring only worked with **real disk paths from the Go backend**, but opening a folder used the browser File System Access API, which only provides file handles — no `C:\...` paths. So `collectMonitoredFolders()` always returned empty and the scan never ran.

### What was wrong
- **Open Folder** used `directoryOpen()` → files got `handle` sources, not `path` sources
- **Drag-and-drop** used HTML5 DnD → virtual paths like `/FolderName/file.trc3`
- Folder scanning (`ScanFolderChanges`) never had a real directory to watch

### What was fixed

1. **Open Folder (desktop app)** — now uses a native Go folder picker (`PickFolder`) and loads via `ReadPaths`, giving real Windows paths.

2. **Drag-and-drop (desktop app)** — Wails `OnFileDrop` is enabled again, which provides full filesystem paths. HTML5 drop is skipped in the desktop app to avoid duplicate handling.

3. **Monitored directory tracking** — `monitoredDirectories` is stored when a folder is loaded via Go, so scanning keeps working even when some files come from ZIP extraction.

### How to test

1. **Restart the app** (`p dev`) so the Go changes load.
2. Use **File → Open Folder...** (not a previously opened session).
3. Confirm monitoring is on (it auto-enables for folders).
4. Add or remove a `.trc3` file in that folder.
5. Within the poll interval (default 5s), you should see:
   - Green/red row highlights in the file list
   - `+N` / `-N` badges in the header

**Important:** If the folder was opened before this fix, close it and reopen via **Open Folder** so paths are registered correctly. Drag-and-drop onto the window also works now in the desktop app.
