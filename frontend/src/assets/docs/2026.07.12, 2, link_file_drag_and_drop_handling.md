## Behavior

When a drag-and-drop lands on the main window and the **only** dropped item is a `.lnk` shortcut:

1. **The web side decides** whether it's a "Link File" (extension check).
2. If a backend exists → the call is **redirected to Go**, which extracts the shortcut's target path and opens the referenced file/folder through the existing load pipeline.
3. If it's a **web-only** build (no Go component) → a message is shown via `notice.info` (the app's `notice.information`-style mechanism).

Anything else (regular files, folders, multi-file drops) keeps its existing behavior untouched.

## Changes

**Web-side detection**
- `frontend/src/workers-client/file-utils.ts` — added `isLinkFile(file)` (checks `.lnk`), exported alongside `isOurFile`/`isTrc3File`/`isZipFile`.
- `frontend/src/components/ui/local-ui/6-dnd/8-dnd-atoms.tsx` — at the top of the drop handler, `getSingleDroppedLinkFile()` runs synchronously during the drop event; if it finds a single `.lnk`, it either routes to Go (`asyncOpenLinkFile`) or shows the notice when no backend is present.

**Frontend → Go bridge**
- `frontend/src/wails/is-wails.ts` — added `OpenLinkFile` to the `BackendApp` type.

- `frontend/src/store/traces-store/8-0-load-files-from-paths.ts` — added `asyncOpenLinkFile(file)` (reads the `.lnk` bytes, sends them base64-encoded to Go) and refactored the shared result handling into `handleReadPathsResult`.
- Updated the checked-in Wails bindings (`wailsjs/go/backend/App.js` / `App.d.ts`).

**Go backend**
- `backend/files.go` — `App.OpenLinkFile(lnkData []byte)` resolves the shortcut target, then reuses `ReadPathsFromDisk` (so a `.lnk` pointing to a `.trc3`/`.zip` file *or* a folder is handled identically to a normal drop, including the "unsupported file" path).
- `backend/link_windows.go` — resolves the shortcut via the Windows Shell COM API (`IShellLinkW` + `IPersistFile`), writing the received bytes to a temp `.lnk` and calling `GetPath`.
- `backend/link_other.go` — non-Windows stub returning an error.

A caveat worth flagging: HTML5 drops don't expose a filesystem path, so Go resolves the shortcut from the dropped `.lnk` **bytes** (via a temp file + COM). This correctly handles shortcuts to files and folders; shortcuts to non-filesystem/virtual targets (e.g. Control Panel) have no path and will surface as a failure the user is notified about.

## What changed

### Open Folder (File menu)
When the Go backend exists, **Open Folder** now uses a native Windows folder picker (`App.OpenFolder` → `runtime.OpenDirectoryDialog`) instead of `directoryOpen`. That bypasses the WebView2 restriction entirely.

In web-only mode, if the browser throws a folder-access error (`SecurityError`, `NoModificationAllowedError`, or a message about system files), the app shows:

> This application cannot open that folder.

via `notice.info`.

### Folder drag-and-drop
When the Go backend exists:
- **Wails `OnFileDrop`** handles drops and passes real filesystem paths to `ReadPaths` — folders with system files work because Go reads them directly.
- The HTML5 drop handler is skipped for file processing (overlay feedback still works).

In web-only mode, if folder enumeration fails with a restricted-access error, the same `notice.info` message is shown.

### Go backend additions
- **`App.OpenFolder()`** — native folder picker + existing `ReadPathsFromDisk` pipeline.
- **`.lnk` path resolution in `ReadPaths`** — so shortcuts dropped via native `OnFileDrop` also resolve correctly.
